任务目标

在 TEdit pure-editing strength control 主线内，完成一个 6-run 最小实验矩阵，用最少改动区分以下三类根因：

当前问题是否主要来自 loss 重心过于 order-first，导致 weak/medium/strong 虽可排序但 gap 偏小。
当前问题是否主要来自 strength_scalar 标尺过窄，导致条件动态范围不足。
当前问题是否主要来自 strength 分支控制权不足，即 projector / modulation 有响应，但最终输出幅度仍被主干或 final pipeline 压平。

本阶段不碰 forecast revision 外层，只在 TEdit pure-editing train/data/eval 内完成根因验证与最小重构。这和现有 phase-1 边界一致。

Run 1：Mild gap-first loss shift

目的：先验证是不是单纯 supervision 偏保守。

只改

TEdit-main/configs/synthetic/finetune_strength_trend_family.yaml

改法

在不改结构的前提下，把当前 loss 重心从 order-first 往 gap-first 推一点：

gain_match_loss_weight = 当前值的 2.0x
family_relative_gain_loss_weight = 当前值的 2.0x
monotonic_loss_weight = 当前值的 0.7x
constant_gain_penalty_weight = 当前值的 1.5x
minimum_family_gain_std = 适度上调
其他保持不动

这一步直接对应“重点押到 gain_match + family_relative + family_std，monotonic 留着兜底”的建议。

预期

如果 final_strong_minus_weak_mean 明显上升而 bg_mae 没明显恶化，说明主因偏 loss。

Run 2：Aggressive gap-first + margin stretching

目的：验证是不是训练目标本身允许模型用过小 gap 过关。

只改

仍然只改 finetune_strength_trend_family.yaml

改法

在 Run 1 基础上继续：

family_relative_margin_scale 设为 > 1，建议先试 1.25
gain_match_loss_weight 可升到 baseline 的 3.0x
family_relative_gain_loss_weight 可升到 baseline 的 3.0x
monotonic_loss_weight 不再继续降

这一步对应你贴出的 required_gap = clamp(target_gap * family_relative_margin_scale, ...) 这条“强度拉伸阀门”。

预期

如果 gap 仍明显上不去，说明不是纯 loss 问题。

Run 3：Scalar rescaling only

目的：验证是不是 strength_scalar 动态范围过窄。

只改
TEdit-main/data/discrete_strength_family.py
如有必要，配套数据构造脚本
改法

回到 baseline loss，不要叠加 Run 2。
只把三档 strength 的 scalar 标尺从“编号式”改成更接近 target gain 比例的标尺。

优先建议：

若当前近似 {0,1,2} 或 {1,2,3}，先改成 {0.0, 0.5, 1.0}
若 family 内 target gain 比例更接近 1:2:4，可试更拉开的映射

同时确保：

diagnostics 不再硬编码只认 [0,1,2]
family 排序逻辑基于 strength_scalar
strength_label 只保留为兼容入口，真实幅度语义以 strength_scalar + target_gain 为中心。
预期

如果不改结构就能把 gap 拉开，问题主要是 scalar 设计。

Run 4：Projector consumes scalar

目的：验证“scalar 只在末端乘一次”是不是主要瓶颈。

改文件
models/conditioning/numeric_projector.py
models/diffusion/diff_csdi_multipatch.py
models/diffusion/diff_csdi_multipatch_weaver.py
改法

现在已有判断是：projector 最后层初始化很小，modulation 最后层也很小，而且 delta_gamma/delta_beta 后面又乘了一次 strength_scalar，最终只是在 base gamma/beta 上做 residual 修正。

本 run 只做最小结构改动：

让 StrengthProjector 直接接收并融合 strength_scalar
_build_strength_condition() 不再把 projector 调用时的 strength_scalar 留空
residual block 末端那次乘 scalar 暂时保留，但减弱其主导地位，避免完全改成另一条机制

目标是把 strength 从“末端音量旋钮”变成“条件空间连续编码”。这正是你贴的建议里最值得先做的小结构升级。

额外要求

在 probe/eval 中补输出：

projector output norm vs scalar
projector pairwise L2 vs scalar
modulation delta norm vs scalar
Run 5：Beta-only / beta-upweight ablation

目的：验证真正有效的控制通道是不是 beta。

改文件
diff_csdi_multipatch.py
diff_csdi_multipatch_weaver.py
二选一

不要两种都同时上。

方案 A：beta_only

保留 delta_beta
暂时关闭 delta_gamma

方案 B：beta_upweight

gamma/beta 都保留
但 beta 分支 multiplier > gamma 分支 multiplier

原因是当前 modulation 形式里，beta 更像位移/偏置，通常比 gamma 的缩放更直接影响编辑幅度。你贴的建议里也明确更看好 beta 通道。

预期

如果这一步明显改善 final gap，说明不是 projector 没表达出来，而是注入后真正能推主干的通道太弱。
